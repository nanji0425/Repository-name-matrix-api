# 管理员后台数据面板设计

日期：2026-07-14

## 目标

在管理员后台新增一个“数据面板”入口，让管理员能快速看到平台今日与累计运营数据，而不需要分别进入用户、日志、订单等页面手工核对。

第一版聚焦“运营总览”，不做复杂 BI 系统。面板要回答四个问题：

1. 今天有没有新增用户、调用和充值。
2. 当前平台总用户、总调用消耗规模是多少。
3. 今天调用是否正常，失败/异常是否需要关注。
4. 哪些模型消耗最多，方便运营和模型管理判断。

## 页面入口

- 后台侧边栏新增菜单：`数据面板`
- 推荐路由：`/panel/dashboard` 或沿用项目后台路由习惯的管理员路径。
- 仅管理员可访问；普通用户不可见、不可访问。

## 第一版指标

核心卡片：

- 今日新增用户数
- 总用户数
- 今日调用次数
- 今日消耗 Tokens
- 总消耗 Tokens
- 今日消费金额
- 今日充值金额
- 今日失败/异常调用次数

辅助模块：

- 近 24 小时调用趋势
- 热门模型 Top 5：按调用次数和 Tokens 展示
- 最近异常摘要：余额不足、模型不存在、请求失败、超时等类型
- 今日用户行为摘要：创建 API Key 用户、有调用用户、充值用户等

## 数据口径

- “今日”按服务器当前时区自然日计算。
- 用户数来自用户表。
- 调用次数和 Tokens 来自消费/调用日志表。
- 消费金额按平台展示额度/金额口径换算，优先复用现有 quota 显示逻辑。
- 充值金额来自充值订单表，统计支付成功的订单。
- 失败/异常调用来自日志状态或错误字段，不把健康接口当作页面可用性指标。

如果某个数据表或字段在当前 New API 版本中不存在，第一版允许该指标显示 `0` 或 `暂无数据`，但 API 响应结构必须稳定，避免页面报错。

## 后端接口

新增管理员接口，建议：

`GET /api/admin/dashboard/metrics`

返回结构建议：

```json
{
  "success": true,
  "data": {
    "today_new_users": 0,
    "total_users": 0,
    "today_requests": 0,
    "today_tokens": 0,
    "total_tokens": 0,
    "today_quota": 0,
    "today_topup": 0,
    "today_failed_requests": 0,
    "hourly_requests": [],
    "top_models": [],
    "error_summary": [],
    "user_activity": {}
  }
}
```

接口必须做管理员权限校验。查询失败时返回明确错误，不让前端无限加载。

## 前端展示

页面采用草图中的“运营总览”布局：

- 顶部标题：数据面板
- 第一屏 8 个指标卡片
- 下方两列布局：调用趋势 + 热门模型
- 再下方：异常摘要 + 用户行为摘要
- 页面提供刷新按钮
- 加载中显示骨架屏或简洁 loading
- 接口失败显示错误提示和重试按钮，不能白屏或无限加载

用户可见文案全部使用平台自身口径：Matrix API 是服务提供方，不出现内部供应链、成本或后台计价配置类口径。

## 测试与验证

实现前先添加可失败的 QA/测试，覆盖：

- 管理员数据面板入口文件或路由存在。
- 页面包含核心指标文案。
- API client 使用管理员数据面板接口。
- 用户可见代码不出现禁用口径词。
- 后端接口或统计 SQL 保持稳定字段名。

实现后验证：

- 本地 QA 通过。
- 前端构建通过。
- 后端相关 Go 测试或编译检查通过。
- 页面接口失败时不会无限加载。

## 不包含在第一版

- 不做复杂日期范围筛选。
- 不做 CSV 导出。
- 不做多维钻取详情页。
- 不做利润/成本公开展示。
- 不对普通用户展示该页面。

这些可以作为第二阶段“分析中心”扩展。
